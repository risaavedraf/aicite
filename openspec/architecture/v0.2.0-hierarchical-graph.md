# v0.2.0 — Hierarchical Graph Architecture

## Status: Proposed

This document describes a major architectural improvement for the Cite CLI: replacing the flat vector store with a hierarchical graph structure. This addresses the core issue of low similarity scores caused by large, noisy chunks.

## Problem statement

### Current state (v0.1.0)

The current architecture is a flat vector store:

```
Document → Chunk (500-1000 chars) → Embedding
```

**Issues**:
- Chunks are large (500-1000 chars) and contain mixed topics
- Cosine similarity scores are low (0.62-0.69) because the query matches only a small portion of the chunk
- Result: `insufficient_context` even when the relevant information is present
- No way to navigate or organize information semantically

### Observed behavior

```
Query: "How are passwords validated?"
Chunk: "The API gateway routes requests... JWT tokens... Passwords must be 12+ chars... Logging uses ELK..."
cosine(query, chunk) = 0.65
                       ↑ "Passwords" matchea, pero el resto "diluye" el score
```

### Root cause

The problem is not cosine similarity or the need for sigmoid normalization. The problem is architectural: chunks are too large and contain too many ideas.

## Proposed architecture

### Hierarchical structure

```
Document (raíz)
  └── Topic (sección semántica)
       └── Concept (unidad de conocimiento)
            └── Chunk (texto atómico, 30-200 chars)
```

**Example**:

```
Document: architecture.txt
│
├── Topic: "API Gateway"
│   └── Concept: "Routing"
│       └── Chunk: "The API gateway routes all external requests to internal microservices" (70 chars)
│
├── Topic: "Authentication"
│   ├── Concept: "JWT Tokens"
│   │   ├── Chunk: "JWT tokens with 15-minute expiry" (30 chars)
│   │   └── Chunk: "Refresh tokens valid for 7 days" (30 chars)
│   │
│   └── Concept: "Password Policy"
│       └── Chunk: "Minimum 12 characters with uppercase, lowercase, digit, special" (60 chars)
│
├── Topic: "Database"
│   ├── Concept: "PostgreSQL"
│   │   └── Chunk: "PostgreSQL with read replicas for high availability" (50 chars)
│   │
│   └── Concept: "Failover"
│       └── Chunk: "Automatic failover within 30 seconds" (35 chars)
│
└── Topic: "Logging"
    └── Concept: "ELK Stack"
        └── Chunk: "Structured JSON logs shipped to centralized ELK stack" (55 chars)
```

### Benefits

| Aspect | v0.1.0 (flat) | v0.2.0 (hierarchical) |
|---|---|---|
| Chunk size | 500-1000 chars | 30-200 chars |
| Noise per chunk | High | Low |
| cosine scores | 0.62-0.69 | ~0.85+ |
| Navigation | Search only | Zoom in/out by level |
| Relationships | None | Cross-document semantic links |
| Result kind | frequent `insufficient_context` | accurate `context` |

## Design decisions

### Decision 1: Topic identification strategy

**Choice**: Hybrid approach — Headings + Semantic clustering + CLI correction

**Process**:

1. **Headings extraction** (primary)
   - Parse markdown headings (`##`, `###`) as topic/concept boundaries
   - Works well for structured documents
   - Fast, deterministic, no API calls

2. **Semantic clustering** (cross-document connections)
   - After generating embeddings, cluster similar chunks across documents
   - Identifies relationships between documents
   - Example: "JWT tokens" in architecture.txt connects to "Token security" in security-policy.txt

3. **CLI correction** (agent refinement)
   - Agent can review and correct topic names after ingestion
   - Commands: `topics list`, `topics rename`, `topics move-concept`, `topics split`, `topics merge`

**Rule of root connection**:
- Chunks can be connected to other documents via semantic clustering
- But the primary identity always comes from the original document
- A chunk's "address" is always: `Document > Topic > Concept > Chunk`

```
Document A: architecture.txt
  └── Topic: "Authentication"
       └── Concept: "JWT Tokens"
            └── Chunk: "JWT tokens with 15-min expiry"  ← PRIMARY

Document B: security-policy.txt
  └── Topic: "Security"
       └── Concept: "Token Security"
            └── Chunk: "Tokens signed with RS256"

Semantic link: JWT ↔ RS256 (via clustering)
But each chunk maintains its root: A/auth/JWT vs B/security/RS256
```

### Decision 2: Chunk strategy

**Choice**: Sentence-based with merge (30-200 chars)

**Algorithm**:

```rust
fn chunk_smart(text: &str) -> Vec<String> {
    // 1. Split by sentences
    let sentences = split_sentences(text);
    
    // 2. Merge short sentences (<30 chars) with the next one
    let mut chunks = Vec::new();
    let mut current = String::new();
    
    for sentence in sentences {
        if current.len() + sentence.len() < 200 {
            current.push_str(&sentence);
            current.push(' ');
        } else {
            if !current.is_empty() {
                chunks.push(current.trim().to_string());
            }
            current = sentence;
        }
    }
    
    if !current.is_empty() {
        chunks.push(current.trim().to_string());
    }
    
    chunks
}
```

**Rationale**:
- Respects natural sentence boundaries
- Keeps chunks small (30-200 chars)
- Merges very short sentences to avoid tiny fragments
- One chunk ≈ one idea

**Alternatives considered**:
| Strategy | Pros | Cons |
|---|---|---|
| Fixed size | Simple, predictable | Cuts mid-sentence |
| Sentence-based | Natural boundaries | Variable size |
| Concept-based | Semantically pure | Complex, needs LLM |
| Sliding window | No context loss | Duplication |

### Decision 3: Retrieval strategy

**Choice**: Chunk-first, topic as context

**Process**:

```
Query: "How long do JWT tokens last?"
        │
        ▼
┌─────────────────────────────────────────────────────┐
│  Step 1: Search CHUNKS (vector similarity)          │
│  ─────────────────────────────────────────────────  │
│  - "JWT tokens with 15-min expiry"     → 0.95      │
│  - "Refresh tokens valid for 7 days"   → 0.91      │
│  - "All data encrypted AES-256"        → 0.45      │
└─────────────────────────────────────────────────────┘
        │
        ▼ (top 5 chunks)
┌─────────────────────────────────────────────────────┐
│  Step 2: Enrich with TOPIC CONTEXT                  │
│  ─────────────────────────────────────────────────  │
│  Chunk: "JWT tokens with 15-min expiry"             │
│    ├── Topic: "Authentication"                      │
│    ├── Concept: "JWT Tokens"                        │
│    └── Document: "architecture.txt"                 │
└─────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────────┐
│  Step 3: Return to agent                            │
│  ─────────────────────────────────────────────────  │
│  {                                                  │
│    "text": "JWT tokens with 15-min expiry",         │
│    "score": 0.95,                                   │
│    "context": {                                     │
│      "topic": "Authentication",                     │
│      "concept": "JWT Tokens",                       │
│      "document": "architecture.txt",                │
│      "breadcrumb": "arch.txt > Auth > JWT"          │
│    }                                                │
│  }                                                  │
└─────────────────────────────────────────────────────┘
```

**Why chunk-first**:
- Most queries ask for specific details ("how long do tokens last?")
- Small chunks match precisely
- Topic context is metadata enrichment, not a filter

**Performance**:
- Vector search: ~500ms (same as v0.1.0)
- Context enrichment: ~5ms (simple SQLite queries)
- Total: ~505ms — nearly identical to current

### Decision 4: Backward compatibility

**Choice**: Keep flat search as fallback

```bash
# Hierarchical search (new default)
cite context "query"

# Flat search (legacy fallback)
cite context "query" --flat
```

**Implementation**:

```rust
fn retrieve(query: &str, config: &RetrievalConfig) -> Vec<Chunk> {
    if config.use_hierarchy && db.has_hierarchy() {
        // New: hierarchical retrieval
        retrieve_hierarchical(query, config)
    } else {
        // Legacy: flat retrieval
        retrieve_flat(query, config)
    }
}
```

## Schema changes

### New tables

```sql
-- Topics: semantic sections within documents
CREATE TABLE topics (
    topic_id TEXT PRIMARY KEY,
    document_id TEXT NOT NULL REFERENCES documents(document_id),
    name TEXT NOT NULL,
    summary TEXT,
    embedding BLOB,
    chunk_count INTEGER DEFAULT 0,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Concepts: atomic knowledge units within topics
CREATE TABLE concepts (
    concept_id TEXT PRIMARY KEY,
    topic_id TEXT NOT NULL REFERENCES topics(topic_id),
    name TEXT NOT NULL,
    summary TEXT,
    embedding BLOB,
    chunk_count INTEGER DEFAULT 0,
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Cross-document semantic links (from clustering)
CREATE TABLE semantic_links (
    link_id TEXT PRIMARY KEY,
    source_chunk_id TEXT NOT NULL REFERENCES chunks(chunk_id),
    target_chunk_id TEXT NOT NULL REFERENCES chunks(chunk_id),
    similarity_score REAL NOT NULL,
    link_type TEXT NOT NULL DEFAULT 'semantic', -- 'semantic', 'reference', 'related'
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### Modified tables

```sql
-- Add hierarchy references to chunks
ALTER TABLE chunks ADD COLUMN concept_id TEXT REFERENCES concepts(concept_id);
ALTER TABLE chunks ADD COLUMN topic_id TEXT REFERENCES topics(topic_id);

-- Add indexes for hierarchical queries
CREATE INDEX idx_chunks_concept ON chunks(concept_id);
CREATE INDEX idx_chunks_topic ON chunks(topic_id);
CREATE INDEX idx_topics_document ON topics(document_id);
CREATE INDEX idx_concepts_topic ON concepts(topic_id);
CREATE INDEX idx_semantic_links_source ON semantic_links(source_chunk_id);
CREATE INDEX idx_semantic_links_target ON semantic_links(target_chunk_id);
```

## CLI commands

### Topic management

```bash
# List all topics
cite topics list

# List topics for a specific document
cite topics list --doc architecture.txt

# Show topic details with concepts
cite topics show topic_abc123

# Rename a topic
cite topics rename topic_abc123 "API Design"

# Move a concept to a different topic
cite topics move-concept concept_xyz789 --to-topic topic_def456

# Split a topic (create new topic with some concepts)
cite topics split topic_abc123 --concepts concept1,concept2 --name "Networking"

# Merge topics
cite topics merge topic_abc123 topic_def456 --name "Infrastructure"

# Show semantic links for a chunk
cite links show chunk_abc123

# Rebuild hierarchy (re-ingest with hierarchy)
cite ingest doc.txt --rebuild-hierarchy
```

### Enhanced search

```bash
# Search with breadcrumb context (default)
cite context "query"

# Search flat (legacy)
cite context "query" --flat

# Search within a specific topic
cite context "query" --topic "Authentication"

# Search within a specific document
cite context "query" --doc architecture.txt

# Search topics only (high-level)
cite search "query" --level topic

# Search concepts only
cite search "query" --level concept
```
